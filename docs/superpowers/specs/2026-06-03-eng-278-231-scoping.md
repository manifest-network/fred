## The bug (ENG-278, recap)

`internal/backend/docker/reconcile_custom_domain.go`: the diff loop mutates `prov.Items[idx].CustomDomain` **under `provisionsMu`**, then **unlocks** (~the `b.provisionsMu.Unlock()` before the `b.Restart(...)` call), then calls `b.Restart`, which **re-acquires** the lock and re-reads `b.provisions[uuid]` to snapshot `prov.Items` (`restart_update.go` ~line 82). Between the unlock and `Restart`'s re-lock, `recoverState` (runs every `reconcile_interval`) can swap the provision struct — rebuilt from container labels = the **old** domain — into `b.provisions[uuid]` (`recover.go` merge preserves only `Provisioning`/`Restarting`/`Updating`, **not `Ready`**; the lease is still `Ready` in this window because status flips to `Restarting` only inside the actor, after the unlock). `Restart` then snapshots the swapped struct → recreate with the old domain → the staged change is dropped for that tick. Self-healing (next tick re-detects drift) but causes a redundant container restart (brief downtime) + delayed cert issuance.

Verified pre-existing: the mutate→unlock→`Restart` pattern is in base `3e54bd7` (pre-ENG-266); ENG-266 only changed the computed `desired` value (the DNS gate), not the locking structure. ENG-264 made the reconcile→Restart path actually fire for single-image leases, so the latent window is now reachable.

## Design questions to answer (the deliverable)

1. **What moves into the actor?** The custom-domain *apply* (mutate `prov.Items`) + the redeploy must happen atomically under the actor's serialization, so there's no off-actor mutate-then-unlock-then-Restart. Options:
   - Extend `RestartRequestedMsg` (or a new `SetCustomDomainMsg`) to carry the desired per-item custom_domain values; the actor's entry action applies them to `prov.Items` AND drives the replace — all serialized against `recoverState` (which takes `provisionsMu`, and the actor's apply also takes it / runs in the serialized transition).
2. **Does the rollback CAS survive?** If the apply is actor-serialized, is the ServiceName+value-CAS rollback still needed, or does it simplify away?
3. **Does `recoverState` need any change?** Ideally the actor-serialization alone closes the window (no risky `recoverState` change). Confirm.
4. **Keep the ENG-266 DNS gate as-is:** `desired` is still computed before the lock (the precompute); only the *apply + restart* moves into the actor. The gate's asymmetric semantics (never tear down emitted, never gate a clear) must be preserved.
5. **The concurrency regression test:** interleave `ReconcileCustomDomain` with `recoverState` (a concurrent struct-swap) and assert the recreate uses the **new** domain — and that it fails without the fix. Recall the ENG-266 lesson: a green `-race` on synchronous tests proves nothing; this test needs the real concurrent interleaving.
